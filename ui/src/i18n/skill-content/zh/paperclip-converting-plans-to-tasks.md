# Paperclip — 把计划拆成任务

把计划转化为可执行 Paperclip 工作的配套技能。本技能**不规定**计划的结构 —— 你可以采用任何最适合工作和用户偏好的格式。它告诉你**如何**把计划翻译成 issue,让 Paperclip 的其余机制为你所用。

关于**记录计划的具体操作**(以 `plan` 为 key 的 issue 文档、评论链接、审批门控、重新指派给谁),请参考 `paperclip` 技能的"Planning"章节。本技能讲方法论,不讲 API 表面。

## 当你被要求做计划时

- **深入计划。** 把你掌握的真实细节都写下来:目标、约束、未知数、成功标准、风险。浅层计划下游变成返工 —— 受派人只能根据他们能读到的内容采取行动。
- **了解你的团队。** 在指派任何工作之前,先查阅公司里的 agent 及其专长(汇报关系、角色描述、过往工作)。如果有更合适的 agent 存在,不要默认把工作留给自己;不要指派给一个你没核实过的名字。
- **按专长分派。** 把每一块工作交给最相关的 agent。如果没有合适的人,把这件事说出来 —— 招聘、工具、外部依赖、董事会决策 —— 而不是用模糊的指派来掩盖空缺。
- **承担责任。** 专长匹配是双向的:当**你**就是某项工作的最佳人选时,把它指给自己,而不是反射性地下放。不要为了减负而推卸。
- **善用依赖树。** Paperclip 的执行器会自动启动任何没有未关闭 blocker 的已指派任务。把每个具体交付物表达成 issue,通过 `blockedByIssueIds` (而不是"被 X 阻塞"这样的散文)接入真实的 blocker。当上游 `done`,下游会自动唤醒。
- **先排序,再并行。** 按真实依赖排序,而不是按个人偏好。图中独立的分支应当并行启动。和人类不同,大多数 agent 允许并发运行,所以你可以把并行工作分给同一个 agent。
- **够用即止。** 计划存在是为了解锁执行,而不是替代执行。如果下一步小而清晰,直接做或者让计划独立成立。把计划本身再计划一遍,或者把一个 agent 在拆分时间内就能完成的工作再切碎,都是拖延 —— 出货第一。

## 发布计划前的快速核查清单

- [ ] 详细到受派人不需要回头追问。
- [ ] 每个具体交付物都是一个 issue (或被命名为已知的后续项)。
- [ ] 每个 issue 都有刻意的、按专长匹配的受派人 —— 而不是默认指派给规划者本人。
- [ ] 每个 issue 的真实 blocker 都通过 `blockedByIssueIds` 声明。
- [ ] 独立分支可以并行启动。
- [ ] 缺口(缺技能、缺招聘、缺决策、缺外部输入)被显式提出,而不是被隐藏。

## 这个技能不是什么

- 不是计划模板。任何格式都行 —— 散文、提纲、表格、RACI、甘特图,只要合适。
- 不局限于软件开发。同一套规则适用于市场、研究、运营、设计、招聘、财务等领域。
- 不能替代 `paperclip` 技能里的计划机制。两者一起用。
